# Scripter

AI-powered video script editor that runs multiple LLM evaluations in parallel and synthesizes them into a master summary. Write markdown scripts, get scored feedback from 5+ models simultaneously, see a unified summary with score comparisons, and get concrete text-level edit suggestions with inline editor highlights.

## Tech Stack

- **Framework**: Next.js 16 (App Router, TypeScript, React 19)
- **UI**: shadcn/ui (New York style, zinc theme, CSS variables), Tailwind CSS v4, `lucide-react` icons
- **Editor**: `@tiptap/react` WYSIWYG with `@tiptap/markdown` for serialization, plain `<textarea>` for raw markdown mode
- **Markdown rendering**: `react-markdown` + `remark-gfm` + `rehype-raw` + custom callout plugin (`src/lib/markdown/callout-plugin.ts`)
- **AI/LLM**: Vercel AI SDK v6 (`ai` package) with `@openrouter/ai-sdk-provider`
- **Database**: SQLite via `drizzle-orm` + `better-sqlite3`, stored at `data/scripter.db`
- **State**: `zustand` with `persist` middleware
- **Layout**: `react-resizable-panels` v4 for split view
- **Theming**: `next-themes` (default dark)

## Architecture

### Data Flow

```
User writes script in Tiptap WYSIWYG editor (or raw markdown mode)
  → Clicks "Evaluate" → runs all active models in parallel via /api/evaluate
  → Each model streams markdown text to its tab (ReadableStream on client)
  → Server-side: after stream ends, captures structured tool call data (scores/criteria JSON)
  → Persists both markdown result + structured JSON to SQLite
  → After all models finish, /api/summarize builds a score comparison table from structured data
  → Summary agent streams a synthesis with score tables, consensus, contradictions
  → Summary agent also calls submit_suggestions tool with concrete text edits
  → /api/summarize captures tool call, validates originalText exists in script, persists to DB
  → Client fetches suggestions when summary completes → shows as inline highlights + floating popover
  → User clicks highlight → popover with diff/reasoning → accepts (text replaced) or dismisses
```

### Key Design Decisions

- **DB lazy-init via proxy**: `src/lib/db/index.ts` exports `db` as a Proxy that calls `getDb()` on first access. Avoids build-time SQLite lock contention.
- **`ensureDb()` in API routes**: Every API route calls `ensureDb()` from `src/lib/db/init.ts` which runs migrations + seeds on first request.
- **OpenRouter as single gateway**: One `OPENROUTER_API_KEY` env var routes to Claude, GPT, Gemini, Grok, DeepSeek. Provider logic in `src/lib/llm/providers.ts`.
- **Streaming**: `streamText()` → `toTextStreamResponse()` returns a plain text stream. Client reads via `ReadableStream` reader. NOT `toDataStreamResponse()`.
- **Structured tool calls**: Models output markdown text AND call a `submit_evaluation` tool with structured JSON (scores, criteria, priorities). The tool call is captured server-side after the stream ends — no client changes needed. Stored in `evaluations.structured_result` column.
- **Fallback for tool calls**: If a model doesn't support or skips the tool call, `structured_result` stays null and the summary falls back to parsing raw markdown.
- **Agentic suggestions**: The summary agent calls a `submit_suggestions` tool alongside its markdown output. The `/api/summarize` route captures the tool call in a background Promise (same pattern as evaluate), validates each suggestion's `originalText` exists in the script, and persists to the `suggestions` table. Client fetches via `GET /api/suggestions`. Previous suggestions are cleared on re-evaluation.

### Database Schema

6 tables in `src/lib/db/schema.ts`:

| Table | Purpose |
|---|---|
| `scripts` | Script metadata (title, context, current version) |
| `script_versions` | Versioned script content with word counts |
| `evaluations` | Model evaluation results (markdown + structured JSON), status tracking |
| `evaluation_profiles` | Reusable evaluation prompts with criteria weights |
| `custom_models` | User-added models (OpenRouter or Ollama) |
| `suggestions` | AI-generated text edit suggestions (originalText, suggestedText, reasoning, priority, status) |

Migrations in `drizzle/` folder, generated by `drizzle-kit generate`. Run automatically on first request via `drizzle-orm/better-sqlite3/migrator`.

## Gotchas & Lessons Learned

### Vercel AI SDK v6 API

- **Tool definition uses `inputSchema`, NOT `parameters`**. The `tool()` helper from `ai` wraps a `Tool<INPUT, OUTPUT>` object where the schema field is `inputSchema: FlexibleSchema<INPUT>`. Passing `parameters` silently fails type checking in some cases or throws "no overload matches".
- **Tool call results use `.input`, NOT `.args`**. `StaticToolCall` has `{ toolName, input }` not `{ toolName, args }`. When extracting tool call data: `submitCall.input`, not `submitCall.args`.
- **Use `toTextStreamResponse()`**, not `toDataStreamResponse()`. The client reads a plain text ReadableStream. `toDataStreamResponse()` returns a different wire format.
- **`result.toolCalls` is a promise** that resolves after the full stream completes. Await it after `result.text` resolves.

### Tailwind CSS v4

- Uses `@plugin` syntax in globals.css (e.g., `@plugin "@tailwindcss/typography"`) instead of v3's `tailwind.config.ts` plugins array.
- `@custom-variant dark (&:is(.dark *))` replaces the v3 darkMode config.
- There is no `overflow-wrap-anywhere` utility class — it doesn't exist. Use plain CSS in globals.css instead: `overflow-wrap: anywhere`.

### Tiptap Editor

- Must set `immediatelyRender: false` in `useEditor()` to avoid SSR hydration mismatches with Next.js.
- Table extensions (`@tiptap/extension-table`, `table-row`, `table-cell`, `table-header`) use **named exports**, not default exports.
- Markdown round-trip: load with `editor.commands.setContent(md, { contentType: "markdown" })`, save with `editor.getMarkdown()`.
- External content sync: skip the `onUpdate` callback when setting content programmatically via a `skipNextUpdate` ref.

### react-resizable-panels v4

- Uses `orientation` prop, NOT `direction`. `<ResizablePanelGroup orientation="horizontal">`.
- To prevent content overflow in split view, add `className="overflow-hidden"` to `ResizablePanel` and the `ScrollArea` wrapper.

### Prose/Markdown Wrapping in Constrained Containers

Setting `overflow-wrap: anywhere` on `.prose` alone isn't enough — child elements (p, li, td, blockquote, a, code) don't inherit it. Must target them explicitly in CSS:
```css
.prose p, .prose li, .prose td, .prose blockquote { overflow-wrap: anywhere; word-break: break-word; }
.prose a, .prose code:not(pre code) { word-break: break-all; }
```
Also need `min-w-0` on the prose container (forces flex children to shrink below content size).

### Drizzle Migrations

- Don't manually create migration SQL files + journal entries. Use `npx drizzle-kit generate` which creates the SQL, snapshot JSON, and journal entry together.
- The migrator reads from `drizzle/meta/_journal.json` to find which migrations to run. Each entry needs a matching snapshot file in `drizzle/meta/`.
- SQLite ALTER TABLE only supports ADD COLUMN — no DROP or RENAME in migrations.

### Structured Evaluation Tool Calls

The `submit_evaluation` tool has no `execute` function — it's a schema-only tool. The model calls it to submit structured data, but the server doesn't execute anything. The tool call args (now `.input`) are extracted from `result.toolCalls` and stored as JSON.

The summary route (`/api/summarize`) enriches client-provided evaluations by querying the DB for `structured_result` data, matching on the full text result. This avoids needing the client to know about structured data at all.

### Suggestions System

Same schema-only tool pattern as `submit_evaluation`. The `submit_suggestions` tool is defined in `summarize.ts` — the summary agent calls it after writing its markdown summary. The route captures the tool call in a background Promise and persists valid suggestions to the `suggestions` table.

Key design details:
- **Validation**: Each suggestion's `originalText` is checked against the script content via `scriptContent.includes(s.originalText)`. Hallucinated text is silently discarded.
- **Cleanup**: `deleteSuggestionsForVersion()` is called before persisting new suggestions to prevent stacking from re-runs.
- **Race condition**: Client retries once after 2s if the initial fetch returns empty (the background Promise may still be running).
- **Staleness**: `useSuggestions.getSuggestionsWithStaleness()` checks if `originalText` still exists in the current editor content. Stale suggestions show a warning badge and only offer dismiss (no accept).
- **Inline display**: Suggestions appear as colored highlights in the Tiptap editor via ProseMirror DecorationSet. Clicking a highlight opens a `SuggestionPopover` floating card with diff view, reasoning, and accept/dismiss buttons. No separate Suggestions tab.

### Tiptap Suggestion Highlight Plugin

The suggestion system uses a ProseMirror Plugin (not Tiptap Marks) since suggestions are transient UI overlays:
1. `Extension.create()` with `addProseMirrorPlugins()` returning a Plugin with PluginKey
2. Plugin state holds `{ suggestions, activeSuggestionId, decorations: DecorationSet }`
3. Updates dispatched via `tr.setMeta(pluginKey, { type, ... })` from React state
4. `findTextRange()` maps flat text offsets to ProseMirror positions via `doc.descendants()`
5. Exported helpers: `setSuggestions(editor, ...)`, `setActiveSuggestion(editor, id)`, `findSuggestionAtPos(state, pos)`

## Browser Testing Requirements

After implementing any changes that affect the web UI (components, styles, pages, hooks), you **MUST** test with Claude in Chrome (`mcp__claude-in-chrome__*` tools) before considering the work complete:

1. **Start the dev server** if not already running (`npm run dev`)
2. **Navigate** to the app (`mcp__claude-in-chrome__navigate` to `http://localhost:3000`)
3. **Take a screenshot** to verify visual rendering looks correct
4. **Check console for errors** (`mcp__claude-in-chrome__read_console_messages` with `onlyErrors: true`)
5. **Test key interactions** — click buttons, type in inputs, toggle modes, etc.
6. If errors are found, **fix them and re-test** until the page loads cleanly with zero console errors

This is enforced by a Stop hook — you cannot finish a session without browser verification for UI work.
